LBD-2 (f:\CG Game\Bot-show\LBD-2) is a copy of LBD-1 with a changed concept: the bot is **fixed by dragging number "codes" into the correct slots**, not by charging/splitting batteries.

**Part 1 (done as of 2026-06-15):** three draggable code tiles (the whole + two parts) and three slots (big = whole, two small = parts, either order). Kid sorts each code into its correct slot; wrong drop bounces back + red slot flash. All three placed → success (green circuit flow) → concept → dancing bot. Mechanic lives in `js/batteries.js` (rewritten, still exports `window.Batteries`); Screen-4 concept in `js/concept.js`; tiles styled `.code-tile` + `.code-dock` in `css/screen2.css`.

BOTH parts now use codes (`window.STAGES[*].part1` / `.part2` = `{whole, parts:[a,b]}`, read via `window.getCodes(part)`). Part 1 levels: Tut 8→(5,3); 12→(6,6); 20→(9,11); 20→(8,12); 15→(8,7). Part 2 levels: Tut 8→(2,6); 12→(6,6); 16→(7,9); 20→(9,11); 20→(8,12). (`getCounts` removed.)

**Part 2 (top-down split) — DONE.** Same code mechanic as Part 1 but placed top-down: WHOLE into the big slot FIRST, then PARTS into the small slots. `batteries.js` is now a factory `createCodePuzzle(opts)` ({screenId, contentId, connectorsId, getCodes, guideWholeFirst, onSolved}); `concept.js` is `createConcept(opts)`. Part 1 instances live in those files (`window.Batteries`, `window.ConceptScreen`); Part 2 instances live in `part2.js` (Screen 6 puzzle `guideWholeFirst:true`, Screen 8 concept `wholeFirst:true`, "This whole is made of these two parts"). part2.js keeps Screen 5 overcharged intro + Screen 7 celebrate + zoom transitions. Screens 6/8 got the board-fix markup (cover/connectors/sockets, dock on 6). Per-bot interior via `SCHEME` map in main.js setPanelScheme: Part 2 uses white (tutorial) + red/green/teal/yellow (levels) — slot_<scheme>.webp + board fill + connector hue each. Script order: concept.js, batteries.js, part2.js (factory must load before part2 uses it).

Texts: Screen 1 tutorial "Let us start fixing."; Screen 2 "Drag the codes in the correct slots."; Screen 4 "These two parts make this one whole."

**Slot/socket visuals (important):** the panel_*.webp images bake in WIDE navy slots + orange connectors (sized for the old battery rows). The codes only need narrow slots, so screen-2/4 now COVER that baked central region with a `.slot-cover` patch (flat board colour — `--board-fill` set per scheme in `main.js BOARD_FILL`, sampled from each panel) and draw narrower CSS `.socket` elements + an SVG `.connectors` layer on top. The old `panel--big/small-left/small-right` overlays, `slot-glow-*` SVGs, and `#charge-fx` were REMOVED from screen-2/4 (still present in Part-2 screens 6/8). Success glow = `.socket.is-charged` + `.connectors.is-green/.is-flowing`. Code blocks = `.code-tile`: in the DOCK / while dragging it's a full block — cream box (#ebddc2) + GREY border (#5f5e5e, no shadow) + navy number. On drop into a slot the tile gets `.in-slot` (box+border transparent → ONLY the number shows, light cream #ebddc2, slightly LARGER font 4.3cqw, on the navy slot).

Connectors use **`connector.webp`** (converted from `connector.svg`, a glowing copper tube + metal end-nodes + centre junction box). `.connectors` is a div with 3 `<img class="connector connector--dl/dr/h">`, each positioned by centre + rotated to its segment (CSS in screen2.css). Recoloured per bot via `filter: hue-rotate(var(--conn-hue))` — `--conn-hue` set in `setPanelScheme` (`CONN_HUE`: orange 0, gold 12, blue 188, purple 250, pink 300 deg). On solve `.connectors.is-green .connector` hue-rotates to green. Connectors use NATURAL proportions (`height: auto` — no stretch); each width = the span so the ends JUST TOUCH the slots (horizontal 25.94% small-to-small; diagonals run CORNER-to-CORNER — big slot's bottom corner ↔ small slot's facing/inner top corner, width ~12.5% so the end nodes JUST TOUCH the corners (no overlap, no gap)). Horizontal has a reduced `height: 2cqw` (thinner than its natural ~47px). `.connectors` is z5 (above sockets) so the ends stay visible. Diagonals read a bit thinner than the horizontal (inherent: single thin connector.webp asset + short diagonal span) — only a dedicated diagonal asset would fully equalize. A `.conn-flow` SVG (3 dashed paths along the segments) inside each `.connectors` div animates the travelling current pulse when `batteries.js` adds `.is-flowing` on solve (green via `.is-green`). (Old `--slot-accent` var removed.) `batteries.js` toggles `.in-slot`: added in `placeInSlot`, removed in `sendHome` + on `pointerdown` (so the whole block drags). Concept tiles are built with `.in-slot` (always number-only in their slots).

Slots use **`slot.svg`** art (cream frame + orange ring + navy interior + metal tabs), recoloured per bot: `slot_<scheme>.webp` generated by replacing slot.svg's `#FF8A00` with `SLOT_ACCENT[scheme]` (main.js: orange #ff8a00, gold #e8a30c, blue #2f6fe0, purple #9a45cf, pink #ec3f8f). The `.socket` elements are now `<img>` (height auto, aspect 182:163); `setPanelScheme` swaps their src AND sets `--slot-accent` (which also colours the `.connectors`). `.socket.is-charged` = green drop-shadow glow. `.code-deco` removed. Geometry centres unchanged (big 965,268; small 629,622 / 1302,621); only widths shrank.

The bottom **number tray** is `Number_tray.webp` (converted from `Number_tray.svg` — same art for EVERY bot, colour never changes; `.code-dock` is now an `<img>`). The tray webp was REGENERATED WITHOUT the cream home-pads (removed the 6 pad rects from Number_tray.svg) so codes sit directly on the navy bar — needed so a placed code's faded copy reads as a faded block on NAVY (matching the user's ref), not cream-on-cream. Home centres `DOCK_X=[816,960,1103]`, y=950. Dock order is shuffled each load (`shuffleDock`, never fully asc/desc). To regenerate the webp from an SVG without sharp: render via Playwright (screenshot the SVG `<img>` with omitBackground for alpha, then re-encode the PNG through a canvas `toDataURL('image/webp')` — drawing an SVG directly to canvas taints it).

**Tutorial guide (currentLevel===1 only, `batteries.js`):** instead of the old 3× ghost demo, a step-by-step guide walks ONE code at a time — glow the code (`.code-tile.is-target`) + its target slot (`.socket.is-cue`, white/gold pulse) and loop a ghost of that drag (`ghostLoop`) until the child places it; only the cued code is draggable. Order: the two PARTS → small slots, then WHOLE → big. `guideStep()` advances on a correct drop / re-cues on a wrong drop. Once the child GRABS the cued code (`guideEngaged`), the ghost stops and does NOT replay for that code (glow stays); it resumes for the next code. A placed code leaves a faded WHOLE-block copy in its tray slot (`.code-tile.dock-faded`, `addFade`/`removeFade`) = the normal block (cream fill + grey #5f5e5e border + navy number) at uniform `opacity: 0.2`. Ghost loops use a `ghostGen` token (bumped in `abortHint`) so a superseded loop can't resurrect and demo into an already-filled slot. Levels (currentLevel===2) keep the 12s idle `ghostRun` nudge, no guide.
